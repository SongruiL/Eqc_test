# 设计规格：算子注册表 + 求值器

> 状态：已评审定稿（2026-06-17 确认）
> 范围：EQC 作为「数学模型开发工具」的地基重构。**不含** GP、优化算法、完整量纲系统（量纲为后续单独 spec，本 spec 预留接口）。

## 0. 背景与目标

### 当前问题
EQC 里一个「原语算子」的定义被拆散在 ~6 处，彼此手动同步、已经出现漂移：

- `src/ast/expr.rs`：`Expr` 枚举变体（AST 节点）
- `src/sexpr/converter.rs`：算子名（含别名）→ `Expr`，并检查参数个数
- `src/ast/expr.rs`：`to_python()` / `to_rust()` / `to_latex()` 三个 ~360 项的大 `match`
- `substitute()`、类型检查等再各处理一遍

由此产生的真实语义缺陷（已抽查确认）：
- **`sign`**：Rust 生成 `.signum()`（在 0 处返回 ±1），Python 生成 `np.sign`（在 0 处返回 0）→ 同一算子跨目标不一致，且偏离数学定义 `sgn(0)=0`。
- **`mod` / `rem`** 被合并为同一运算，但数学模与程序余数对负数结果不同。
- **`gcd` / `lcm`** 定义在 `f64` 上，实现却强转 `i64`，对连续量静默截断。

### 目标
1. **算子单一真相源**：每个算子只定义一次（语义 + 各目标代码模板 + 元数据），所有消费者从中派生。
2. **求值器**：对 AST 直接数值求值，作为注册表的第一个消费者；为后续 GP 的 fitness 评估、人工调试提供「即时算数」能力。
3. **钉死边界语义**：把上述模糊点一次性定义清楚，处处一致。

### 验收靶子
`examples/greenhouse.sexpr`（温室微气候-作物响应模型，5 个耦合算子，覆盖 加减乘除 / `exp` / `max` / 嵌套除法）。

---

## 1. 总体架构

```
                 ┌─────────────────────────────┐
                 │     算子注册表 OperatorSpec   │  ← 单一真相源
                 │  名称+别名 / 参数个数 / 类别   │
                 │  求值函数 eval               │
                 │  代码模板 rust/python/latex  │
                 │  定义域与语义说明            │
                 │  〔预留〕量纲规则            │
                 └──────────────┬──────────────┘
        ┌───────────────┬───────┴───────┬───────────────┐
        ▼               ▼               ▼               ▼
   converter        to_rust /        求值器          类型/参数
  (名称→AST)        to_python /      Expr::eval       个数检查
                    to_latex
```

注：`if / sum / product / piecewise / lambda` 是**特殊形式**（绑定变量 / 控制流），不是普通 n 元函数，不进注册表，由 AST 节点专门处理。注册表只覆盖「纯函数式」算子。

## 2. 算子注册表

### 2.1 一个条目（OperatorSpec）应包含
- `id`：规范名（如 `sec`）
- `aliases`：别名（如 `secant`）
- `arity`：参数个数（固定 n / 可变 / 标记为特殊形式）
- `category`：分类（算术 / 三角 / 特殊函数 …）
- `eval`：求值函数 `fn(&[f64]) -> Result<f64, EvalError>`
- `codegen`：各目标的生成模板（rust / python / latex）
- `domain`：定义域与边界语义说明（人读 + 机器可校验的要点）
- `dimension`〔预留，本期不实现〕：量纲传播规则

### 2.2 实现风格（**待决策，见 §7**）
- **A. 数据表**：`OperatorSpec` 结构体 + 函数指针 + 模板字符串，放进一张 `HashMap<&str, OperatorSpec>`。改动小、上手快。
- **B. trait + 每算子一个 impl**：类型最强，但样板代码多。
- **C. 声明宏**：每个算子写一条紧凑声明，宏自动展开出枚举变体 + converter 分支 + 各 codegen 分支 + eval 分支。**最彻底地「定义一次」**，但前期投入大、宏较复杂。

**决定**：先用 **A**（数据表）把「求值 + 元数据」收拢成注册表（迁移成本最低），把现有的 codegen 大 `match` **逐个算子**搬进来；评估顺利后再考虑 **C** 用宏消灭最后的重复。**分期推进，不一次性重写 350KB 的 `expr.rs`。**

## 3. 求值器

### 3.1 形态
树遍历解释器：`Expr::eval(&self, env: &Env) -> Result<f64, EvalError>`。不经过「生成代码→编译」，给输入即出数值。Rust 树遍历对中小公式是微秒级，起步足够；不够再加缓存 / 编译成字节码（**不提前过度设计**）。

### 3.2 求值环境 Env
- `名称 → f64` 的查找（变量 + 参数）。
- 作用域栈：用于 `Sum` / `Product` / `Lambda` 的循环/绑定变量。
- 缺失变量 → `EvalError::UndefinedVar`。

### 3.3 派发
- **叶子**（Const/Pi/E/Var/Param）：直接取值。
- **特殊形式**（If/Sum/Product/Piecewise/Lambda）：专门逻辑（短路、循环累加/累乘、分段选择）。
- **注册表算子**：求值各子节点 → 查注册表 → 调 `spec.eval`。

### 3.4 错误与非有限值策略（**待决策，见 §7**）
- `EvalError`：`UndefinedVar` / `ArityMismatch` / `DomainError{op}` / `Unsupported{op}` / `NonFinite`…
- 结构性问题（未定义变量、参数个数错）→ 一律 `Err`。
- 数值定义域问题（`ln` 负数、除零）→ **决定：默认严格 `Err`，另提供 `strict` 模式开关**（将来 GP 期切到 NaN 传播→惩罚）。

### 3.5 特殊函数后端
核心算子（四则、三角、exp/log…）用标准库 `f64`；特殊函数（gamma/bessel/erf/分布…）复用 Cargo 里已有的可选依赖 `statrs` / `puruspe`，**与 codegen 同样按 feature 门控**；求值器未实现的算子 → `Err(Unsupported)`，**显式失败、不静默**。

### 3.6 一致性保证（白送的回归测试）
属性测试：对随机输入，`expr.eval()` 的结果应当等于「`to_rust()` 生成的代码编译运行」的结果（在算子定义域内）。这能自动抓出求值器与代码生成的任何分歧。

## 4. 语义规范（一次性钉死，处处一致）

| 算子 | 当前问题 | 拟定规范 |
|------|----------|----------|
| `sign` | Rust/Python 在 0 处不一致 | 统一为数学定义 `sgn(0)=0`，修正 Rust 分支 |
| `mod` / `rem` | 被合并 | 拆成两个算子：`mod`=数学模（结果随除数符号，floored），`rem`=程序余数（随被除数符号）；停止互为别名 |
| `gcd` / `lcm` | f64 静默截断 | 要么仅允许整型参数（校验），要么移出连续核心；明确文档 |
| `div` 除零 / `ln` 负数 / `pow` 负底 | 未定义 | 在 §3.4 策略下显式定义 |

本表 = 算子语义的权威来源，求值器与所有 codegen 都遵守。

---

## 5. 迁移路线（分期，每期结束 `cargo test` 全绿）

- **Phase 0**：本 spec 评审通过；确认 `greenhouse.sexpr` 为验收靶子。
- **Phase 1（求值器最小可用）**：定义 `OperatorSpec` / `Env` / `EvalError`；实现 `greenhouse.sexpr` 用到的核心 ~40 算子的 `eval`；加 eval≈to_rust 一致性测试；能对 greenhouse 给一组输入算出 `Pn` 等输出。
- **Phase 2（codegen 接入注册表）**：把 rust/python/latex 生成对核心算子改为从注册表派生，消除多份真相。
- **Phase 3（扩展覆盖）**：求值器按 feature 扩到特殊函数；语义规范（§4）全面落地。
- **Phase 4（量纲接口）**：注册表加「量纲规则」槽位 → 启动量纲检查器（单独 spec）。

---

## 6. 验收标准
1. `greenhouse.sexpr` 每个算子都能被求值器算出数值，且与生成 Rust 的运行结果一致。
2. 给定输入（Tmax/Tmin/RH/I…）能算出 VPD、Pn 等输出。
3. 0 errors；§4 语义点全部有定义并被测试覆盖。

---

## 7. 决策记录（2026-06-17 已定）
1. **注册表实现风格** → ✅ 先用**数据表**，跑顺后再考虑用宏彻底去重（§2.2）。
2. **求值错误策略** → ✅ **默认严格 `Err` + `strict` 模式开关**（§3.4）。
3. **语义定案** → ✅ 采纳 §4：`mod`/`rem` 拆分、`sign(0)=0`、`gcd/lcm` 限整型。
4. **是否对齐同事后端解释器** → ✅ **不对齐**。该平台定位是面向非数学用户的「数学低代码平台」（类 Dify），而 EQC 面向数学专业用户。**EQC 的注册表/求值器即语义权威**，自行实现。EQC 的操作端/平台**后续另行开发**，本期不做，专注把 EQC 这个建模工具打扎实。
