# EQC Studio 前端 v2 重构规格

> 状态：**定稿**（2026-06-25，与首席科学家逐条敲定）。技术栈 spike 已实测通过；#1 Svelte / #2 提交构建产物 / #3 增量 / #4 IA 初稿 / #5 编辑器+LLM / #6 ts-rs 均已采纳。下一步 P1。
> 关联：现 `src/serve_assets/studio.html`（v1，单文件 vanilla JS，约 1300 行）、`docs/spec-gp-studio.md`、Studio server-first 架构（`eqc serve`，EQC 持有事实、契约只增不改）。

## 1. 背景与目标

studio.html（v1）是多个 arc 堆出来的单文件：结构图 / 仿真轨迹 / 情景探索 / 决策优化 / 标定 / 耦合 / GP / 园区录入 / 看懂卡 / 处理区管理。能用、被验证，但：单文件过长、全局函数+全局状态、手拼 HTML 字符串、无类型、无组件、专家视图是一长条往下滚。

**野心**（首席科学家定）：这是多年演进的项目，要做到 **① 浏览器内模型编辑器 ② LLM 流式问答 ③ 重排成工作区**，且持续精修。v1 的技术债会在这些复杂功能上集中爆发。

**两类用户**：园区员工（录田间数据、看「看懂卡」、简单）；首席科学家（结构/仿真/优化/标定/GP/编辑，全套）。

**目标**：换更可维护、可安全重构、利于做复杂交互的技术栈 + 重排信息架构 + 视觉品质，**不破坏地基原则**。

## 2. 立场 / 不变的原则

- **EQC 始终是唯一权威**：Rust 算一切（sim/optimize/GP/SVG/MathML/JSON），前端只消费 `/api/*` 契约、只渲染。**换技术栈与这条正交**——spike 已证明框架应用消费的是同一批契约，边界没动。
- **契约只增不改**：新功能=新端点 / 契约新字段，老前端不受影响。
- **自包含离线二进制**：构建产物仍 `include_str!` 进 `eqc.exe`，运行时不需要 node。
- **增量、不 big-bang**：v2 与 v1 并存，逐工作区搬迁、到平价再切主路由——绝不一次性重写一个能跑的东西。

## 3. 技术栈（决定：Svelte + Vite + TypeScript + singlefile）

- **Svelte 5**（runes 响应式）：编译期把组件编译成精简 vanilla JS，**运行时体积最小**、最适合 singlefile/离线；无虚拟 DOM 开销。
- **Vite**：构建/开发服务器（dev 有 HMR）。
- **TypeScript**：拿类型对着契约写，编译器抓字段拼错/漂移。
- **vite-plugin-singlefile**：把 JS/CSS 全内联进**一个 HTML**。

**spike 实测（2026-06-25，`frontend/`）**：`npm install`（npmmirror+清代理，48 包 9s）→ `npm run build`（vite v6.4.3，806ms）→ **41 KB 单 HTML**（gzip 16 KB，零外链）→ `include_str!` 进 eqc.exe → `/spike` 路由 served、消费 live `/api/models`+`/api/model`。**所有顾虑（除「多一步轻量构建」）均消解。**

> 备选已评估并否决：Vue/React（运行时更大、singlefile 不如 Svelte 利落）；纯 vanilla 模块化（解决不了类型/组件复用，C 类功能仍痛）。

## 4. 构建集成（推荐：提交构建产物作为 served 资产，cargo build 不需 node）

**问题**：`npm run build` 怎么接进 `cargo build`，又不让「clone 后必须装 node 才能编 eqc」。

**推荐方案 = 提交 singlefile 产物（与 v1 的 studio.html 同模型）**：
- Vite `build.outDir` 直接产出到 `src/serve_assets/studio_v2.html`（singlefile）。
- 该 HTML **提交进 repo**，`serve` 用 `include_str!("serve_assets/studio_v2.html")`。
- 前端**源码**（`frontend/src/*`）也在 repo；改前端 → `cd frontend && npm run build` 重新生成 `studio_v2.html` → **连同源码一起提交**。
- **效果**：`cargo build` / CI / clone **永不需要 node**（产物已在 repo），部署模型与今天**完全一致**（一个二进制、include_str）。

**为何不选另两条**：
- `build.rs` 自动跑 npm：把 Rust 构建耦合到 node + 代理 + PATH，脆、慢（每次 cargo build 触发）、CI 复杂 → 否。
- 纯脚本（`just build`）：仍要人记得先 build 前端 → 不如「产物在 repo」省心。

**「产物入 git」顾虑**：moot——v1 的 studio.html 本就是「手写资产入 repo、serve include_str」，这只是把「手写」换成「构建生成」。产物是文本、约 40–100 KB，diff 噪音可接受（或 `.gitattributes` 标 `-diff`）。

> 约定：改 `frontend/src/` 后**必须** `npm run build` 并提交再生成的 `studio_v2.html`；CI 可加一步「build 后 `git diff --exit-code studio_v2.html`」防忘提交（后续）。

## 5. 迁移策略（增量，逐工作区搬迁）

- **P0（已完成）**：spike——`frontend/` 骨架 + 构建管线 + `/spike` 路由验真。
- **P1 外壳 + 1 个工作区**：搭 v2 **应用外壳**（顶栏 + 左导航 + 工作区路由），先搬一个真实工作区（建议「结构」或「仿真」）到 `/v2`。**studio.html 仍是默认 `/`**。
- **P2…Pn 逐工作区搬迁**：仿真 / 优化 / 标定 / GP / 园区，一个一个搬到平价（与 v1 对应面板功能对齐），都挂 `/v2`。
- **P-final 切换**：`/v2` 达全面平价 → 主路由 `/` 切到 v2，旧 studio.html 暂留 `/legacy`，稳定一段后删。
- 每阶段：`npm run build` → 提交产物 → 两配置 `cargo test` + node 校验 + live 核端点（沿用现惯例）。
- **C 类功能**（编辑器/LLM）作为独立阶段，可在核心搬迁后或穿插进行。

**好处**：任何时刻都有一个能跑的前端（v1 或 v2）；回归被隔离在「正在搬的那个工作区」；可随时停在中途不影响线上。

## 6. 信息架构（#4 · 初稿，待微调）

**外壳 = 顶栏（全局）+ 左导航（工作区）+ 主内容区**。杀掉「一长条往下滚」，改成**一次一个任务的工作区**。

```
┌──────────────────────────────────────────────────────────────────────┐
│ EQC Studio   [模型▾ 草莓S8]  [处理区▾ 1]        〔园区|专家〕 ●已连接  〔问AI〕│ ← 顶栏(全局,常驻)
├────────┬─────────────────────────────────────────────────────────────┤
│ 结构    │                                                             │
│ 仿真 ●  │                  主工作区（当前导航项内容）                   │
│ 优化    │                                                             │
│ 标定    │      （「结构图」可作共享上下文：部分工作区左右分栏，         │
│ 进化    │        左结构图 + 右当前任务）                               │
│ 编辑    │                                                             │
│────────│                                                             │
│〔问AI〕 │                                                             │
└────────┴─────────────────────────────────────────────────────────────┘
```

- **顶栏（全局、跨工作区常驻）**：标题 / **模型选择器**（工作区花名册）/ **处理区选择器** / **模式切换（园区·专家）** / 连接状态 / **问AI**（LLM 抽屉开关）。模型+处理区是横切选择，提到全局，所有工作区共享。
- **左导航 = 工作区**（专家模式）：
  - **结构** Structure：Forrester/DAG 图（布局/粒度/缩放/专注/节点交互）+ 二维公式。
  - **仿真** Simulate：整季轨迹图 + 变量选择 + 情景探索器（参数/初值滑块实时重算）。
  - **优化** Optimize：决策优化（+ 耦合条目时的耦合仿真/优化）。
  - **标定** Calibrate：参数标定 + 可辨识性（用本处理区观测）。
  - **进化** GP：现 GP 面板全套（靶点多选→Pareto→候选详情→采纳）。
  - **编辑** Edit（C，新）：模型源码编辑器 + live 校验 + 结构预览。
- **园区模式**：外壳同构但**导航精简 + 放大**——只 **看懂 / 录入 / 标定**（看懂卡、录入网格、本区标定），给园区员工的极简动线。
- **结构图作共享上下文**：很多任务（仿真/优化/GP）都想边看结构边操作 → 这些工作区可左右分栏（左常驻结构图、右任务），或结构图做成可固定的侧栏。具体分栏方式留待实现微调。
- **问AI 抽屉**：右侧滑入的 LLM 对话，跨工作区可用，自动带当前模型 AST + 上下文（选中变量/方程/GP 结果）。

> 这是按我的理解给的**起点**，专为你「改动微调」准备。可调点：导航分组粒度、结构图是否常驻分栏、园区/专家是否改成「角色」、工作区命名、是否要「首页/概览」工作区等。

## 7. C 类新能力（#5 · 推荐）

### 7.1 浏览器内模型编辑器（CodeMirror）
- **编辑** 工作区：CodeMirror 6（YAML 模式）载入模型源码；改动防抖 → POST 校验 → 错误下划线 + 结构预览实时刷新。
- **新端点**：`GET /api/source?model=`（返回 `.eq.yaml` 原文）、`POST /api/validate`（收文本 → 解析+校验 → 返 `{ok, errors[], 可选 structure_svg}`）。Rust 仍是校验权威，编辑器只递交文本。
- **写回谨慎（同 GP 采纳精神）**：v1 = 编辑 + live 校验 + 预览 + **下载**（不自动写盘）；v2 = 带确认/备份/diff 的受控保存端点 `POST /api/source`（后续）。

### 7.2 LLM 流式问答（问AI）
- **右侧抽屉对话**，跨工作区。**新端点** `POST /api/llm`：收 messages + **当前模型上下文**（AST/契约 + 选中变量/方程/GP 结果）→ **流式**（SSE/chunked）回 Claude 应答。模型上下文注入 → 答案 grounded 在「这个模型」，不是泛泛而谈。
- **后端**：这是少数**必须新增后端逻辑**处（LLM 代理 + API key 从 env/config，**绝不放浏览器**）。现手写 HTTP server 是 per-connection 阻塞式，SSE 流式需 chunked transfer——可行但是一项真后端活，单独成阶段。
- **模型选型**：用最新、最能打的 Claude 模型（实现时查 `claude-api` 参考定 model id / 流式 / 工具用法）。可先非流式打通，再上流式。
- **进阶**（后续）：让 LLM 能调 EQC 工具（跑仿真/优化/解释 DAG）= agent 化，复用契约当工具。

### 7.3 结构 diff（后续）
- GP 采纳新方程、或模型版本间差异 → 结构（AST 级）diff 视图。Rust 持有 AST、算 diff，前端渲染。优先级低，列后续。

## 8. 契约类型自动生成（#6 · 推荐：ts-rs）

- 现状（spike）：手写 `frontend/src/contract.ts` 镜像契约——会漂移。
- **推荐 `ts-rs`**：给 `export.rs` 的 `ModelJson`/`ModuleJson`/`VarJson`/`EqJson`/`ParamJson` 等加 `#[derive(TS)]`（dev/feature-gated），一个 `cargo test`/`xtask` 生成 `frontend/src/contract.gen.ts`。
- **效果**：「只增不改的契约」有了**自动同步的类型镜像**——Rust 加字段 → 重新生成 → TS 立刻看到、用错即编译报错。**强化**原则、不削弱。
- **时机**：前端长过 spike 后引入（非阻塞）；引入前先用手写 contract.ts 顶着。

## 10. 风险 / 取舍

- **唯一真新增成本**：前端改动多一步 `npm run build`（806ms）+ 维护 node 工具链（npmmirror 已验证可用；未来可能专开一台项目虚拟机）。
- **迁移期双前端并存**：短期维护成本，但换来零回归风险——值得。
- **LLM 端点**：唯一显著新增后端 + 引入 API key 管理 + 流式——单独成阶段、谨慎做。
- **守住原则**：每个工作区只消费契约；任何「想把逻辑放前端」的诱惑都回到「该不该进 Rust 当权威」来判。
